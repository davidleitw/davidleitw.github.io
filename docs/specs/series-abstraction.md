# Spec:讓「系列」成為網站真正認識的東西

狀態:ready-for-agent
範圍:文章頁系列導覽、系列順序的單一來源、`/posts` 檔案樹的部級巢狀

---

## Problem Statement

站上有兩組文章其實是「一份有順序的長文件」,不是一堆獨立文章:

- **Agentic Design Patterns 繁中翻譯**:1 篇目錄頁 + 38 篇內文(6 篇前言、21 章、7 篇附錄、4 篇結尾),原書分成四大部。
- **Hermes From Zero**:1 篇總覽 + 15 篇 Day 01–15。

但網站只在「文章清單」的層次認得它們。讀者實際打開第 7 章時,頁面上沒有任何東西告訴他:

1. 這是一本書的一部分。
2. 現在讀到第幾篇、總共幾篇、屬於哪一部。
3. 怎麼跳到別章,而不用退回目錄頁再點進來。

更嚴重的是**順序這件事沒有真相來源**,而是散在三個地方,而且其中兩個是錯的:

- 文章頁的上一篇/下一篇是拿**全站文章按發佈時間排序**算出來的。書裡的章節能正常運作,純粹是因為當初把時間戳手動排成遞減(目錄頁 21:00、第 1 章 20:59、第 2 章 20:58⋯⋯獻詞 20:22)。
- 因為前言六篇的閱讀順序(獻詞 → 致謝 → 序 → 思想領袖 → 前言 → 什麼是 Agent)跟時間戳遞減方向相反,**在前言那一段按「下一篇」是往回走的**。
- 最舊的一篇(獻詞)按「下一篇」會**跳出這本書**,連到站上另一篇不相干的舊文。
- 書的四大部結構(第一部:基礎模式⋯⋯)只以純文字手打在目錄頁的 markdown 裡,程式碼完全不知道。
- `postSeries.ts` 裡的 `bookOrder()` 排的不是閱讀順序,是「章節排前面、附錄接著、前言墊後」——那是為了讓 `/posts` 樹狀圖的截斷預覽好看而排的,跟真正的閱讀順序無關。

結果是:只要之後新增一篇文章的發佈時間落在 20:22–21:00 這個區間,或是重新調整任何一篇的日期,整本書的導覽就會散掉,而且不會有任何東西報錯。

## Solution

把「系列」從**列表頁的顯示規則**升級成**網站的一個概念**。

具體來說:

1. 每個系列有一份 **manifest**,用明確的順序列出它包含哪些文章、分成哪幾部。這份 manifest 成為順序與成員資格的唯一真相來源,取代目前靠時間戳與第一個 tag 推斷的做法。

2. 讀者在系列內任何一篇文章的頁面上,標題上方會看到一條**系列標頭**:系列名稱、所屬部別、在整個系列中的位置(例如 `8 / 38`),以及一顆打開**全系列目錄**的按鈕。

3. **上一篇/下一篇改成依 manifest 排**,在系列內顯示為「上一章 / 下一章」,到系列頭尾就停住,不會漏到系列外面的文章。

4. `/posts` 檔案樹裡的系列資料夾,**展開後是依部別的子資料夾**,而不是現在那份順序刻意調過的六筆截斷預覽。

5. **建置時驗證**:manifest 列了但檔案不存在、檔案存在但沒被任何 manifest 認領、一篇文章被兩個系列認領、目錄頁手打的連結跟 manifest 對不起來——這些都在 `pnpm build` 當下就報出來,而不是等讀者踩到。

## User Stories

### 讀者(在系列文章內)

1. 作為從搜尋引擎直接落在第 12 章的讀者,我想在標題上方看到「這是 Agentic Design Patterns 的一部分」,這樣我才知道自己不是讀到一篇孤立的文章。
2. 作為讀者,我想看到自己在整個系列中的位置(例如 `20 / 38`),這樣我才知道還剩多少。
3. 作為讀者,我想看到目前這篇屬於哪一部(例如「第三部:上線前的考量」),這樣我知道它在整本書的哪個階段。
4. 作為讀者,我想點系列名稱就回到目錄頁,這樣我可以看全貌。
5. 作為讀者,我想在不離開目前文章的情況下打開完整章節清單,這樣我可以直接跳到想看的那章。
6. 作為讀者,我想在那份章節清單裡一眼看到自己現在在哪一章(高亮),這樣我不用自己找。
7. 作為讀者,我想在章節清單裡看到哪些我已經讀過(勾記),這樣我知道從哪裡接下去。
8. 作為讀者,我想看到「已讀 12 / 38」這種進度,這樣我有讀完一本書的動力。
9. 作為讀者,我想按「下一章」就真的到下一章,而不是被丟到某篇不相關的舊文。
10. 作為讀者,我想在前言那幾篇按「下一篇」時是往後讀而不是往回讀。
11. 作為讀者,我在系列第一篇按「上一章」時,想回到系列目錄頁而不是掉到系列外。
12. 作為讀者,我在系列最後一篇時,不想看到一個會把我丟出系列的「下一章」。
13. 作為讀者,我想讓上/下一章的按鈕文案講「章」而不是泛泛的 Previous/Next,這樣我知道它在系列脈絡裡動。
14. 作為讀者,打開章節清單時我想直接看到目前章節捲到視野內,這樣在 38 條清單裡我不用自己捲。
15. 作為讀者,我想用 Esc 關掉章節清單,這樣我不用去找關閉按鈕。
16. 作為 Hermes From Zero 的讀者,我想要跟書一樣的導覽體驗,只是單位講「天」不是「章」。
17. 作為讀者,我在讀一般非系列文章時,不想看到任何系列相關的東西,頁面應該跟現在一模一樣。

### 讀者(手機 / 無障礙 / 降級情境)

18. 作為手機讀者,我想要章節清單以全螢幕面板打開,而不是擠在旁邊的窄欄裡。
19. 作為手機讀者,我想要系列標頭在窄螢幕上自動省略次要資訊(部別),避免換行成三行。
20. 作為只用鍵盤的使用者,我想用 Tab 走到章節清單按鈕並用 Enter 打開它。
21. 作為使用螢幕閱讀器的人,我想聽到系列標頭是有意義的導覽區塊,而不是一串散落的文字。
22. 作為開了「減少動態效果」的使用者,我不想要章節清單有位移動畫。
23. 作為關掉 JavaScript 的使用者,我仍然想能打開章節清單並跳章——這部分不該依賴 JS。
24. 作為使用無痕模式(localStorage 被封鎖)的讀者,我不想看到任何錯誤,已讀勾記與進度靜靜消失就好。

### 讀者(在 /posts 檔案樹)

25. 作為瀏覽 `/posts` 的讀者,我想展開書的資料夾後看到四大部的子資料夾,而不是一份看不出結構的六筆清單。
26. 作為讀者,我想看到每一部裡面有幾篇,這樣我知道展開會看到多少東西。
27. 作為讀者,我想讓部與部的展開狀態跟現在的資料夾一樣被記住,這樣我在文章與清單之間來回時不用重展。
28. 作為讀者,我想讓非系列的資料夾(LeetCode、Linux⋯⋯)行為完全不變,因為它們本來就不是有順序的東西。
29. 作為讀者,我想讓書在「最近更新」跟首頁「最新文章」仍然只佔一筆,不要洗版——這是現有行為,不能弄壞。

### 作者(David)

30. 作為作者,我想把一個系列的順序寫在一個地方,這樣我不用同時維護時間戳、目錄頁連結、跟排序函式三份。
31. 作為作者,我想在 manifest 裡新增一章就讓所有導覽自動跟上,不用改任何其他檔案。
32. 作為作者,我想在忘記把新章加進 manifest 時被建置擋下來,而不是上線後才發現那章沒人連得到。
33. 作為作者,我想在 manifest 列到一篇不存在的文章時被建置擋下來。
34. 作為作者,我想在目錄頁手打的連結跟 manifest 順序對不起來時被建置擋下來。
35. 作為作者,我想調整某篇文章的發佈日期而不弄壞任何導覽順序。
36. 作為作者,我想把某一章暫時設成 draft,而它自動從系列目錄與上下章鏈中消失,不留下死連結。
37. 作為作者,我想在未來新增第三個系列時只要寫一份 manifest,不用碰任何元件。
38. 作為作者,我想讓系列成員資格不再依賴「第一個 tag」,因為我可能會為了 SEO 調整 tag 順序。
39. 作為作者,我想要一組跑得很快的測試涵蓋這些邊界情況,這樣我改動時心裡有底。

### 未來的維護者

40. 作為未來讀這份程式碼的人,我想讓系列邏輯集中在一個純函式模組裡,不散在各個 `.astro` 檔案的 frontmatter 中。
41. 作為未來的維護者,我想讓系列邏輯不依賴 `astro:content` 的型別,這樣測試不需要跑整個 Astro。

## Implementation Decisions

### Seam(測試接縫)

**整個功能只開一道接縫:series core 模組的公開 API。** 它是純函式,輸入是一個最小的文章形狀(`href` / `title` / `date`),輸出是系列索引與單篇文章的系列脈絡。所有 `.astro` 檔案都只是薄薄的轉接層——把 `CollectionEntry<"blog">` 對映成那個最小形狀,然後把結果攤成 HTML。

刻意不用 `astro:content` 的型別當輸入,因為那是虛擬模組,會逼測試載入整個 Astro 執行環境。用結構型別(structural type)代表測試就是一般的 vitest,沒有任何框架設定。

### Series manifest

新增一個 series 資料模組,每個系列一份 manifest,外加一份登記表把它們收在一起。manifest 的形狀:

```ts
type SeriesPart = {
  /** 部別標題;省略代表這個系列不分部,標頭不顯示部名 */
  title?: string;
  /** 這一部包含的文章 href,依閱讀順序 */
  hrefs: string[];
};

type SeriesDefinition = {
  id: string;          // "agentic-design-patterns"
  name: string;        // 顯示用全名
  shortName: string;   // 標頭用短名
  landing: string;     // 目錄頁 href,同時是這個系列的入口
  tag: string;         // /posts 檔案樹用來歸資料夾的 tag slug
  unitLabel: string;   // "章" / "天" — 上下篇按鈕的量詞
  parts: SeriesPart[];
};
```

決定用**完整 href** 而不是相對 slug,因為兩個系列的檔案佈局不同(書是子目錄,Hermes 是平鋪檔名),用 href 就沒有任何拼接規則要記,而且跟目錄頁 markdown 裡已經寫著的東西字面一致,好對照。

書的 manifest 依原書結構分成六部分:前言、第一部到第四部、附錄、結尾。Hermes 只有一部,`title` 省略。

### Series core 模組

取代現有的 `postSeries.ts`。公開 API:

- `buildSeriesIndex(posts)` — 吃全站文章,回傳一個索引:每個系列的完整有序條目(標題從文章 frontmatter 取,不在 manifest 裡重複)、href 到位置的對映、以及驗證診斷。
- `getSeriesContext(index, href)` — 回傳單篇文章的系列脈絡,非系列文章回 `null`。
- `buildFolders(index, posts)` — `/posts` 檔案樹的資料夾,系列資料夾帶部級子結構。
- `collapseRecent(index, posts, limit)` — 維持現有的「系列收合成一筆」行為,只是成員判定改讀索引。

`getSeriesContext` 的回傳形狀:

```ts
type SeriesContext = {
  series: { id; name; shortName; landing; unitLabel; total: number };
  isLanding: boolean;
  position: number | null;   // 1-based;目錄頁為 null
  partTitle?: string;
  prev: { href; title } | null;
  next: { href; title } | null;
  parts: { title?: string; entries: { href; title; isCurrent }[] }[];
};
```

`parts` 帶完整目錄,讓標頭與抽屜共用同一份資料,不用第二次查詢。

保留 `cleanTitle`(把 `Agentic Design Patterns:` 前綴從章節標題剝掉)與 `displayName` / tag 顯示名對照表,那些跟系列抽象無關但仍在用。刪掉 `bookOrder` 與 `dayOrder`——它們的職責被 manifest 完全取代。

### 成員資格與順序

成員資格改由 **manifest 的 href 清單**決定,不再看第一個 tag。`/posts` 檔案樹仍用 tag 歸資料夾(那是顯示分類),但「這是不是一個有順序的系列」改問索引。兩者刻意分開:資料夾是 tag 分群,系列是有序文件。

順序完全由 manifest 決定,發佈時間不再參與系列內排序。

### 文章頁

新增一個 series 導覽元件,放在文章標題卡片的上方,由 `PostDetails` 在有系列脈絡時渲染:

- 一行標頭:系列短名(連到目錄頁)› 部別 · `8 / 38`,以及右側一顆開啟目錄的按鈕,按鈕上顯示已讀進度。
- 目錄面板用原生 `<details>` 實作,不用 JS 也能開合,跟 `/posts` 檔案樹的作法一致。JS 只負責增強:Esc 關閉、開啟時把目前章節捲進視野、套用已讀勾記。
- 面板在窄螢幕佔滿畫面,寬螢幕靠右浮出。刻意**不做常駐側邊欄**:文章頁是置中單欄 `max-w-4xl`(xl 才 `max-w-5xl`),兩側剩下的寬度塞不下 38 條中文章節標題,而抽屜在手機與桌機共用一套實作。

上下篇改成:有系列脈絡時用 manifest 的前後鄰居,按鈕文案改成「上一{unitLabel} / 下一{unitLabel}」;系列第一篇的「上一篇」指向目錄頁;系列最後一篇沒有「下一篇」。沒有系列脈絡時,完全維持現行的全站日期排序行為。

已讀狀態沿用現有的 `readPosts` localStorage 鍵(文章頁已經在寫、`/posts` 已經在讀),不新增儲存格式。

### /posts 檔案樹

系列資料夾展開後改成部級子資料夾(巢狀 `<details>`),目錄頁仍釘在最上面。非系列資料夾維持現有的「六筆 + … N more」。

展開狀態的 sessionStorage 鍵沿用現有的,但值改成可以同時表達資料夾與子資料夾的形式;舊值(單純的資料夾 slug 陣列)要能被安全讀取,讀不懂就當作沒有偏好,回到伺服器預設。

### 驗證

在建置索引時做四項檢查:

| 情況 | 處理 |
|---|---|
| manifest 列的 href 沒有對應文章(檔案不存在、被刪、設為 draft) | 從目錄中略過,印警告,不中斷建置 |
| 文章帶著系列的 tag 但沒被任何 manifest 認領 | 附在該系列最後(不讓文章消失),印警告 |
| 同一個 href 被兩份 manifest 認領 | 拋錯,中斷建置 |
| 目錄頁 markdown 裡的系列內連結,順序或集合跟 manifest 不一致 | 拋錯,中斷建置 |

draft 走「略過 + 警告」而不是拋錯,因為把一章暫時設為 draft 是正常的編輯動作,不該擋住建置。

目錄頁比對的作法:從目錄頁文章的原始 markdown 內文抽出所有指向該系列的連結,依出現順序比對 manifest 的攤平清單。這樣手打的目錄跟 manifest 不可能默默走偏,同時又不必把目錄頁改成 MDX 元件(見 Out of Scope)。

## Testing Decisions

好的測試在這裡的定義:**只測 series core 模組對外的行為**——給定一組文章與 manifest,問它「第 12 章的上一章是誰」「獻詞的下一篇是誰」「這篇屬於哪一部」。不測內部怎麼建 Map、不測元件的 class 名稱、不做快照。

專案目前**沒有任何測試框架**,只有 `astro check` 做型別檢查。這次引進 vitest,理由是系列順序的邊界情況多、而且已經真的壞過一次(前言段的上下篇是反的),而這些邏輯剛好全是純函式,測起來沒有任何框架成本。

測試對象只有 series core 一個模組,涵蓋:

- 章節中段:上一章 / 下一章是 manifest 的鄰居。
- **前言那六篇的順序**(現行 bug 的回歸測試):`introduction` 的下一篇是 `what-makes-an-ai-system-an-agent`,不是 `a-thought-leaders-perspective`。
- 系列第一篇:上一篇是目錄頁。
- 系列最後一篇:下一篇是 `null`,不會外漏。
- 目錄頁本身:`position` 為 `null`、`isLanding` 為真、下一篇是第一篇。
- 非系列文章:回 `null`。
- manifest 列了但文章不存在:該條目從目錄消失,前後鄰居直接相接不留空洞,並回報一筆診斷。
- 文章存在但沒被認領:附在系列尾端,並回報一筆診斷。
- 同一篇被兩份 manifest 認領:拋錯。
- 部別歸屬:第 8 章回報「第二部:進階系統」。
- `collapseRecent`:連續的系列文章折成一筆、非系列文章維持獨立、系列與非系列交錯時不會誤併。
- `buildFolders`:系列資料夾產出部級子結構且總數正確;非系列資料夾維持平坦。
- 發佈時間與 manifest 順序衝突時,以 manifest 為準。

沒有既有測試可以當範例(這是第一組),所以測試風格上刻意保持樸素:寫一個小工廠函式造假文章,`describe` 依 API 分組,斷言只碰回傳值。

`pnpm build` 維持原樣(`astro check && astro build && pagefind`),測試另外掛在 `pnpm test`。

## Out of Scope

- **常駐桌面側邊欄**。版面是置中單欄,兩側寬度不足以放 38 條中文標題;抽屜在手機與桌機共用一套實作,先做抽屜。
- **文章內標題的浮動目錄(heading TOC)**。那解決的是「一篇很長」的問題,跟「一個系列很長」是不同的問題,綁在一起會讓兩邊都難改。之後要做的話是另一個元件。
- **把書的目錄頁改成 MDX 並用元件產生目錄**。技術上可行(MDX 整合已裝好、目錄頁內文沒有會破壞 MDX 的字元),但站上目前一篇 `.mdx` 都沒有,拿一篇 95 行的內容檔當第一個實驗對象不划算。建置時的連結比對已經讓兩邊不可能走偏,所以先不做。
- **自動判定系列**(例如「這個 tag 超過 N 篇就當系列」)。現行的白名單是刻意的:LeetCode 那 14 題不是系列。
- **跨系列導覽**(讀完書接著推薦 Hermes)。
- **伺服器端的閱讀進度**。已讀狀態留在 localStorage,換裝置不同步,這是可接受的。
- **調整任何文章的發佈時間**。這次讓順序不再依賴時間戳,但不去動既有時間戳,避免影響 RSS 與既有連結的排序印象。

## Further Notes

- 這次改動不會動到任何文章內容檔,除非驗證抓到目錄頁與 manifest 不一致。
- 現行「書在最近更新只佔一筆」的行為是刻意設計,不能因為重構而回退;測試裡有涵蓋。
- Hermes 的目錄頁是一張帶每日一句話描述的表格,那些描述不存在於文章 frontmatter,所以它的目錄不適合自動產生,維持手打 + 建置比對。
- 書的 21 章時間戳目前是遞減排列(第 1 章最新),那是為了讓依日期排序的清單讀起來像 1→21。這次之後系列導覽不再依賴這個安排,但時間戳本身保留不動。
